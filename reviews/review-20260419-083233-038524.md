## Medium

### 1. Client upload limits are advertised but not enforced before file processing

- **File path:** `src/app/page.tsx:179`
- **Severity:** medium
- **Category:** Diff
- **Description:** The upload flow advertises a maximum of 5 images and 10MB per image, but the client does not enforce those constraints before creating object URLs and later converting files to base64. Users can select too many images or very large files. In mock mode those files are accepted silently, and in live mode they are fully read into memory before the server rejects them. This creates avoidable memory pressure and inconsistent behavior versus the UI copy.
- **Suggested fix:** Validate selected files before creating object URLs or base64 strings. Enforce accepted image MIME types, maximum count including existing observations, and maximum per-file byte size. Show user-facing validation feedback for rejected files.

### 2. API route parses full JSON body before applying size limits

- **File path:** `src/app/api/analyze/route.ts:14`
- **Severity:** medium
- **Category:** Diff
- **Description:** The route calls `req.json()` before enforcing `MAX_IMAGES` or `MAX_IMAGE_SIZE_BYTES`. An accidental or malicious oversized JSON payload can consume server memory before validation runs. The decoded-size check is useful after parsing, but it does not bound request payload size early enough.
- **Suggested fix:** Add an explicit request-size guard before parsing where possible, such as checking `content-length` and rejecting unknown or oversized bodies. For production use, consider moving large image upload handling to a streaming/upload mechanism with server-side limits.

### 3. API accepts malformed or arbitrary base64 image payloads

- **File path:** `src/app/api/analyze/route.ts:33`
- **Severity:** medium
- **Category:** Diff
- **Description:** The API accepts any string after stripping an optional `data:image/...;base64,` prefix. It does not verify that the payload is valid base64 or that the decoded content is actually a supported image type. Malformed or arbitrary payloads can therefore be forwarded to the VLM backend.
- **Suggested fix:** Validate each image string with a strict accepted data URL/base64 format, decode it with failure handling, verify magic bytes or MIME type against an allowlist such as PNG/JPEG/WebP, and forward only validated raw base64.

### 4. Expensive analysis endpoint has no auth, rate limiting, or origin guardrails

- **File path:** `src/app/api/analyze/route.ts:12`
- **Severity:** medium
- **Category:** Holistic
- **Description:** The API route accepts unauthenticated POST requests that can forward up to five 10MB decoded images to the local VLM. This may be acceptable for a local demo, but structurally it is an expensive compute endpoint with no rate limiting, origin protection, or authorization boundary.
- **Suggested fix:** Document demo-only exposure clearly in `AGENTS.md` and `README.md`. Before deploying beyond localhost, add authentication or local-only guardrails, rate limiting, and request body limits aligned with VLM capacity.

### 5. Home page is both a broad client boundary and a god component

- **File path:** `src/app/page.tsx:1`
- **Severity:** medium
- **Category:** Next.js / Holistic
- **Description:** The entire home route is marked `"use client"` and implements the full workflow in one large component: state management, file handling, mock analysis, API calls, object URL lifecycle, presentation, and result rendering. The route does need client interactivity, but the current boundary is broader than necessary and prevents the static shell from using Server Components by default. It will also become hard for agents or new developers to modify safely as features grow.
- **Suggested fix:** Keep `src/app/page.tsx` as a Server Component and move the interactive upload/analyze workflow into focused client components and/or a hook, such as `ImageUploader`, `FindingsList`, `ActionPlan`, and `useRetailAnalysis()`. Pass only serializable data across the server/client boundary.

### 6. `DirectPlanner` business logic has no automated tests

- **File path:** `src/lib/planner.ts:67`
- **Severity:** medium
- **Category:** Diff / Holistic
- **Description:** `DirectPlanner` defines business-critical ordering by urgency and confidence, summary formatting, fallback behavior, and effort/reasoning mapping, but the project has no tests for it. Regressions would directly affect generated action plans.
- **Suggested fix:** Add focused unit tests for `DirectPlanner.plan()`, including empty findings, mixed urgency ordering, confidence tie-breaks within the same urgency, summary singular/plural text, and fallback behavior for unknown or widened finding types.

### 7. VLM response parser is critical, private, and untested

- **File path:** `src/lib/vlm-client.ts:54`
- **Severity:** medium
- **Category:** Diff / Holistic
- **Description:** `parseFindings` handles untrusted model output, including markdown fences, JSON extraction, type filtering, and confidence clamping. It is private inside `vlm-client.ts` and untested, making future prompt/format changes risky and harder for agents to validate without exercising `RealVLMClient`.
- **Suggested fix:** Export the parser as an internal/testable helper or move it into a pure module such as `src/lib/vlm-parser.ts`. Add tests for valid fenced JSON, malformed JSON, extra prose, non-array JSON, invalid finding types, out-of-range confidence values, missing fields, empty output, and multiple arrays.

### 8. Project lacks a test framework and test scripts

- **File path:** `package.json:5`
- **Severity:** medium
- **Category:** Holistic
- **Description:** The project has no test framework, no test scripts, and no discovered `*.test.*` or `*.spec.*` files. Core logic exists in `src/lib/planner.ts`, `src/lib/vlm-client.ts`, and `src/app/api/analyze/route.ts`, so agents have no fast safety net when changing planning, parsing, or API validation.
- **Suggested fix:** Add a test stack, preferably Vitest for pure logic and API route validation. Start with `DirectPlanner`, VLM response parsing, and `/api/analyze` request validation. Add scripts such as `test`, `test:watch`, and optionally `test:e2e`.

### 9. Agent harness is incomplete

### 10. Environment configuration is scattered and not validated

- **File path:** `src/app/api/analyze/route.ts:7`
- **Severity:** medium
- **Category:** Holistic
- **Description:** Environment configuration is read directly from `process.env` in the route module and passed to `RealVLMClient` without centralized parsing or validation. Documentation mentions optional env vars, but there is no `.env.example`, schema, or startup validation for malformed endpoints or model names.
- **Suggested fix:** Add a central server-only config module, such as `src/lib/server-config.ts`, with explicit parsing or schema validation. Validate `OLLAMA_ENDPOINT` as a URL, validate model names as non-empty strings, document defaults in `.env.example`, and keep this code under a server-only boundary.

### 11. Ollama integration has no timeout or typed response validation

- **File path:** `src/lib/vlm-client.ts:123`
- **Severity:** medium
- **Category:** Holistic / Next.js
- **Description:** `RealVLMClient` directly calls the Ollama REST API via `fetch` with no timeout, retry policy, request aborting, or typed response validation. A hung local model or malformed Ollama response can stall the API route. The route also lacks an explicit execution-duration strategy for hosted Next.js runtimes.
- **Suggested fix:** Add an `AbortController` timeout around the fetch, define and validate a typed Ollama response shape, map integration failures to stable internal error types, and consider exporting a route-level `maxDuration` if deploying to a platform that supports it.

### 12. Object URLs are not cleaned up on component unmount

- **File path:** `src/app/page.tsx:187`
- **Severity:** medium
- **Category:** Diff / Next.js / Holistic
- **Description:** `URL.createObjectURL(file)` is used for image previews, and URLs are revoked when an item is removed or when the user presses reset. If the user uploads files and navigates away while observations remain, the blob URLs can stay allocated for the lifetime of the document.
- **Suggested fix:** Add a `useEffect` cleanup that revokes all current observation URLs on unmount, taking care not to double-revoke URLs in normal remove/reset paths. Alternatively, move object URL management into a small preview component or hook that owns creation and cleanup.
